# OTrap v2.0 — Enterprise OT Deception Platform

Distributed, production-ready ICS/OT honeypot with stateful S7-300/400 emulation,
gRPC-based sensor mesh, and a Next.js SOC management console.

---

## Architecture

```
OT Network                         Management Network
──────────────────                 ──────────────────────────────
OTrap Sensor (Go)   ──gRPC/mTLS──▶  OTrap Manager (FastAPI)
  :102  S7comm                        :8080  REST API + SSE
  :502  Modbus                        :9443  gRPC (sensor mesh)
  :80   HMI HTTP                    PostgreSQL 16
  :443  HMI HTTPS                   Redis 7
                                    Next.js UI  :3000
                                    LLM Engine  :8001 (optional)
```

**Design decisions:**
- Sensors initiate all connections outbound to Manager — no inbound management ports on the sensor
- Per-sensor mTLS certificates issued by an internal CA at join time
- Stateful S7 memory map: attacker writes are readable back, producing a convincing PLC emulation
- CPU STOP returns a plausible ACK (never RST) to maintain deception
- Single-use join tokens with configurable TTL

## Installation

### Step 1 — Clone the repository

```bash
git clone https://github.com/Janberkb/OTrap-v2-ICS-Honeypot-Intelligence-Platform.git
cd OTrap-v2-ICS-Honeypot-Intelligence-Platform
```

### Step 2 — Configure environment

```bash
cp .env.example .env
```

Open `.env` and set at minimum:

| Variable | Description |
|---|---|
| `POSTGRES_PASSWORD` | PostgreSQL password (auto-generated if left as placeholder) |
| `API_SECRET_KEY` | 64-char hex for session signing (auto-generated) |
| `ENCRYPTION_KEY` | 32-char key for AES-256 secrets at rest (auto-generated) |
| `INITIAL_ADMIN_PASSWORD` | First superadmin password (auto-generated) |
| `SENSOR_CERT_ENC_KEY` | 64-char hex for sensor cert encryption (auto-generated) |

> **All secrets marked `CHANGE_ME` are auto-generated by the installer if left unchanged.**
> You can skip manual editing for a quick local start — the installer fills everything in.

**If you plan to deploy sensors on remote hosts**, also set before running the installer:

```dotenv
# The IP or hostname of the management server, reachable from sensor hosts
GRPC_HOST=192.168.1.10
SENSOR_PUBLIC_MANAGER_ADDR=192.168.1.10:9443
```

> The management server must be reachable on TCP/9443 from all sensor hosts.

### Step 3 — Install the Manager

```bash
./scripts/install_manager.sh
# or equivalently:
make install-manager
```

The installer:
1. Verifies Docker + Compose are available
2. Auto-generates any missing secrets in `.env`
3. Checks that ports 8080, 3000, 9443 are free
4. Starts `postgres`, `redis`, `manager`, and `ui`
5. Waits up to 3 minutes for the Manager to become healthy
6. Extracts and persists the auto-generated gRPC CA (`GRPC_CA_KEY_B64`, `GRPC_CA_CERT_B64`) back into `.env` so the CA survives manager restarts
7. Prints admin credentials and the management URL

At the end you will see output like:

```
✓ Manager install complete
  Management UI:  http://localhost:3000
  Manager API:    http://localhost:8080/api/v1
  Admin user:     admin
  Admin password: <generated-password>
```

### Step 4 — Open the Management UI

Navigate to `http://localhost:3000` and log in with the credentials printed by the installer.

- **Superadmin** accounts land on `/admin` (system, users, integrations, audit log)
- **Operator** accounts land on `/dashboard` (live feed, sessions, sensors, health)

## Sensor Deployment

OTrap supports two sensor scenarios:

| Scenario | Use case |
|---|---|
| **Same host** | Local testing — sensor and manager on the same machine |
| **Remote host** | Production — sensor runs on dedicated OT network hardware |

### Scenario A — Same-host sensor (local testing)

The default `docker-compose.yml` includes a local sensor service. After the manager is running:

```bash
# 1. Generate a join token via the UI or CLI
ADMIN_PASS=<your-admin-password> SENSOR_NAME=local-sensor make sensor-token
# The command prints a SENSOR_JOIN_TOKEN value — copy it.

# 2. Add the token to .env
# Edit .env:
#   SENSOR_JOIN_TOKEN=<token printed above>
#   SENSOR_INSECURE_JOIN=true   (already set by installer for local use)

# 3. Start the local sensor
docker compose up -d sensor
```

The local sensor connects to `manager:9443` over the internal Docker `backend` network and joins using the token. Check logs:

```bash
docker logs -f otrap-sensor-1
```

A successful join prints:

```
{"level":"INFO","msg":"Join successful","sensor_id":"..."}
```

### Scenario B — Remote sensor (separate host)

#### B1. Set GRPC_HOST to a reachable IP

The Manager's gRPC port must be bound to an IP the sensor hosts can reach.
**Token generation is blocked when `GRPC_HOST` is loopback** (`127.0.0.1`).

Edit `.env` on the management server:

```dotenv
GRPC_HOST=192.168.1.10                    # management server IP, reachable from sensor host
SENSOR_PUBLIC_MANAGER_ADDR=192.168.1.10:9443
```

Restart the manager to rebind the port:

```bash
docker compose up -d manager
```

#### B2. Build and push the sensor image

The sensor image is **not published to any public registry**. You must build it yourself
and push it to a registry accessible from the sensor host.

```bash
# Single-arch build (common case)
docker build -t <your-registry>/otrap-sensor:latest ./sensor
docker push <your-registry>/otrap-sensor:latest

# Multi-arch build (amd64 + arm64, for ARM industrial hardware)
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t <your-registry>/otrap-sensor:v2.0.0 \
  -t <your-registry>/otrap-sensor:latest \
  --push ./sensor
```

Update `.env` with your image reference:

```dotenv
SENSOR_IMAGE_REF=<your-registry>/otrap-sensor:latest
```

Restart the manager so the new image ref is picked up for generated commands:

```bash
docker compose up -d manager
```

#### B3. Generate an onboarding command

1. Log into `http://<management-server>:3000`
2. Go to **Sensors → Add Sensor**
3. Enter a name (e.g. `ot-segment-a`)
4. Click **Generate**

The UI displays:
- A one-line **Docker run command** — the primary deployment path
- A **join token** (single-use, expires after `JOIN_TOKEN_TTL_HOURS`, default 24 h)
- An advanced **`.env.sensor` snippet** for manual compose deployment

> If `GRPC_HOST` is still loopback, the Generate button returns an error.
> Set a real IP (Step B1) before generating.

Alternatively, from the management server CLI:

```bash
ADMIN_PASS=<your-admin-password> SENSOR_NAME=ot-segment-a make sensor-token
```

#### B4. Run the command on the sensor host

Copy the generated command and run it on the target machine (Docker must be installed):

```bash
docker run -d \
  --name otrap-sensor-ot-segment-a \
  --restart unless-stopped \
  -p 102:102 \
  -p 502:502 \
  -p 80:80 \
  -p 443:443 \
  -v /var/lib/otrap/sensor/certs:/etc/otrap/sensor/certs \
  -e SENSOR_MANAGER_URL=192.168.1.10:9443 \
  -e SENSOR_JOIN_TOKEN=<single-use-token> \
  -e SENSOR_NAME=ot-segment-a \
  -e SENSOR_CERT_ENC_KEY=<64-char-hex-key> \
  -e SENSOR_INSECURE_JOIN=true \
  -e LOG_LEVEL=info \
  <your-registry>/otrap-sensor:latest
```

> `SENSOR_INSECURE_JOIN=true` skips TLS verification for the initial join only —
> the manager's CA is not yet trusted at this point. After a successful join the sensor
> stores a signed mTLS certificate and all subsequent connections are fully verified.

Check that the sensor joined:

```bash
docker logs otrap-sensor-ot-segment-a
# Expected: {"level":"INFO","msg":"Join successful","sensor_id":"..."}
```

Within seconds the sensor appears as **active** on the Sensors page.

#### B5. Advanced manual path (optional)

If you need to deploy via Docker Compose instead of a single `docker run`:

```bash
# On the sensor host — copy docker-compose.sensor.yml from this repo, then:

cat > .env.sensor <<'EOF'
SENSOR_MANAGER_URL=192.168.1.10:9443
SENSOR_JOIN_TOKEN=<token from Manager>
SENSOR_NAME=ot-segment-a
SENSOR_CERT_ENC_KEY=<64-char hex>
SENSOR_INSECURE_JOIN=true
SENSOR_IMAGE_REF=<your-registry>/otrap-sensor:latest
LOG_LEVEL=info
EOF

docker compose -f docker-compose.sensor.yml --env-file .env.sensor up -d
```

> `docker-compose.sensor.yml` uses `network_mode: host` so the sensor binds
> directly to the host's physical interfaces. This is correct for dedicated
> Linux industrial hardware where host networking is available.

## Development Mode

Starts the manager with uvicorn hot-reload, enables FastAPI `/docs`, and exposes Postgres/Redis ports for local tooling:

```bash
docker compose -f docker-compose.yml -f docker-compose.dev.yml up
```

To run the UI natively (faster iteration):

```bash
cd ui
npm install
INTERNAL_API_BASE=http://localhost:8080 npm run dev
# Open http://localhost:3000
```

FastAPI docs available at `http://localhost:8080/docs` in dev mode.

## Security Hardening (Production)

1. **Change all secrets** — never run with `CHANGE_ME` defaults in `.env`
2. **Bind management ports to internal IPs**:
   ```dotenv
   MANAGEMENT_HOST=10.0.0.1
   UI_HOST=10.0.0.1
   GRPC_HOST=10.0.0.1
   ```
3. **Set `SESSION_SECURE=true`** when the UI/API is served over HTTPS
4. **Use strong, unique `POSTGRES_PASSWORD`** — exposed only to the backend network
5. **Rotate sensor tokens** — each token is single-use; revoke unused sensors from the UI
6. **Do not expose port 9443** outside the management network
7. **Sensor ports 102/502/80/443** should only be reachable from OT network segments
8. **Keep `GRPC_CA_KEY_B64` secret** — it is the root signing key for all sensor certs
9. **Use `SENSOR_INSECURE_JOIN=false`** after all sensors have completed their initial join

## Architecture Deviations from Original Spec

| Area | Original | Current | Reason |
|---|---|---|---|
| Sensor language | Python | Go | Native binary, stateful TCP, ARM support |
| Sensor comms | None (monolith) | gRPC + mTLS | Distributed architecture, network stealth |
| UI framework | Previous stack | Next.js 14 App Router | Rebuilt from scratch per spec |
| S7 memory | Stateless | Stateful per-session map | Higher deception fidelity |
| HMI TLS | Manual | Auto-generated self-signed | Operational simplicity |
